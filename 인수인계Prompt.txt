## 복사 시작

당신은 기존 Python 데이터 분석 프로젝트를 인수받는 시니어 데이터 분석가이자 ML 엔지니어입니다. 이 폴더는 **자동차 부품업계 리스크 분석 에이전트** 프로젝트 전체입니다. 먼저 저장소 전체를 조사하고, 아래에 적힌 현재 상태를 실제 파일과 대조한 다음 작업하세요.

### 2. 프로젝트 목적

DART 재무데이터와 ECOS·FRED 거시경제지표를 결합해 자동차 부품 상장기업의 재무 위험을 조회하고 설명하는 프로토타입입니다.

현재 가능한 기능:

- 21개 기업의 2019~2024년 재무 위험지표 조회
- 부채비율, 유동비율, 이자보상배율, 영업이익률, Altman Z-score 계산
- 기업별 연도 추이와 동종기업 중앙값·건전성 순위 비교
- PanelOLS 회귀, XGBoost walk-forward 검증, SHAP 설명
- 과거 관측 범위 안에서 거시변수를 변경하는 모델 민감도 분석
- Gemini 2.5 Flash를 이용한 자연어 질문과 5단 리포트 변환
- FastAPI `/analyze`, `/report`
- Streamlit 6개 탭 대시보드
- Markdown·한글 PDF 리포트

이 프로젝트를 **부도예측, 주가예측, 투자추천, 신용평가 자동화 모델**로 해석하면 안 됩니다. 현재 적절한 명칭은 `공시 기반 재무위험 모니터링·설명 프로토타입`입니다.

분기 손익 교정 규칙:

- DART `fnlttSinglAcntAll`의 `thstrm_amount`는 Q1~Q3 손익계산서에서 해당 3개월 값으로 사용
- Q1~Q3은 추가 차감하지 않음
- 사업보고서 Q4는 연간값이므로 Q1~Q3 합계를 차감해 단일 Q4로 변환
- 기존 잘못된 연속 차감값은 `src/features/repair_quarterly_flows.py`로 역산 복원
- 교정 전 음수 매출 102건이 교정 후 2건으로 감소
- 자본잠식 기업은 음수 부채비율이 건전하게 보이지 않도록 부채비율을 결측 처리하고 별도 위험 경고로 관리

### 4. 현재 모델과 검증 결과

현재 앱에서 사용하는 모델:

- `models/risk_predictor_v2.pkl`
- 로딩 계층: `src/models/risk_model.py`
- 입력: 현재·시차 거시변수와 기업 인코딩
- 부채비율 모델은 로그 학습 후 `expm1` 역변환

최신 동일표본 walk-forward 비교:

| 지표 | 연간 XGBoost R² | 직전연도 기준선 R² | 교정 분기 XGBoost R² | 직전분기 기준선 R² |
|---|---:|---:|---:|---:|
| 부채비율 | 0.554 | 0.854 | 0.626 | 0.724 |
| 유동비율 | 0.858 | 0.913 | 0.818 | 0.793 |
| 이자보상배율 | 0.240 | 0.286 | -0.708 | -0.716 |
| 영업이익률 | 0.242 | 0.514 | 0.029 | 0.058 |
| Z-score | 0.794 | 0.905 | 0.902 | 0.934 |

세부 검증 파일:

- `data/outputs/model_validation_v2.csv`
- `data/outputs/q_model_validation.csv`
- `data/outputs/panel_regression_results_v2.csv`
- `data/outputs/q_panel_regression_results.csv`

중요한 해석:

- 기업별 재무 위험 수준과 지속성 신호는 존재함
- 대부분 지표에서 XGBoost가 단순 직전값 기준선을 이기지 못함
- 교정 분기 유동비율만 기준선을 소폭 상회함
- 높은 R²를 곧바로 우수한 미래예측력으로 표현하면 안 됨
- SHAP beeswarm에서 기업 ID 영향이 거시변수보다 큰 편임
- SHAP은 모델이 왜 해당 출력을 냈는지 설명할 뿐 실제 인과관계를 증명하지 않음
- 금리와 Z-score의 통계적 연관성이 나타나도 거시변수는 같은 시점의 여러 기업에 반복되고 시간 관측치가 적으므로 인과효과로 해석하지 않음
- 시나리오는 `bounded sensitivity`, 즉 관측 범위 내 민감도이며 미래 예측이 아님

### 7. 실행 가능한 환경이라면 검증할 명령

프로젝트 루트에서 실행하세요. Windows PowerShell 기준입니다.

```powershell
python -m pip install -r requirements.txt
python -m unittest discover -s tests -v
```

현재 기대값은 테스트 `7/7 통과`입니다. 실패하면 무조건 코드를 고치기 전에 환경 문제, 패키지 버전, 경로 문제, 데이터 파일 존재 여부부터 구분하세요.

Streamlit:

```powershell
python -m streamlit run app/streamlit_app.py
```

브라우저 주소:

```text
http://127.0.0.1:8501
```

FastAPI:

```powershell
python -m uvicorn app.main:app --reload
```

API 문서:

```text
http://127.0.0.1:8000/docs
```

기존 저장 데이터로 파이프라인을 재생성해야 할 때만 아래 순서로 실행하세요.

```powershell
python -m src.features.repair_quarterly_flows
python src/features/risk_calculator_quarterly.py
python src/features/build_annual_from_quarterly.py
python notebooks/analysis_v2.py

```

`repair_quarterly_flows`는 재실행해도 값이 다시 변하지 않도록 설계됐지만, 실행 전 파일과 변경 범위를 먼저 확인하세요. DART·ECOS·FRED API 재수집은 사용자가 명시적으로 요청한 경우에만 수행하세요.

### Claude

1. Project knowledge나 ZIP 업로드에 코드·문서·CSV를 포함한다.
2. Claude가 코드를 실행할 수 없는 대화 환경이면 정적 코드리뷰와 실행 검증을 구분하게 한다.
3. 결과가 길어지면 먼저 파일 인벤토리와 위험요소를 받고, 다음 메시지에서 세부 파일 검토를 요청한다.

## 업로드 전 보안 체크

- `.env` 제외
- `.streamlit/secrets.toml` 제외
- API 키가 포함된 스크린샷·로그 제외
- Git 기록에 키가 들어갔는지 확인
- 노출된 키가 있었다면 업로드 전에 폐기·재발급
- 회사 내부 데이터가 추가됐다면 외부 AI 업로드 허용 여부 확인
